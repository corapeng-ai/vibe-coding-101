The Problem

Investment Research Is Slow and Fragmented

A retail investor doing proper due diligence spends hours across multiple tools

Manual Process Today
  • 1
    Read macro analysisBloomberg, Reuters, analyst reports — 30+ min
  • 2
    Screen stocksFinviz, Yahoo Finance, filter by fundamentals — 45+ min
  • 3
    Check risk metricsCorrelation, beta, drawdown scenarios — 20+ min
  • 4
    Scan newsEarnings, regulatory, sentiment — 20+ min
  • 5
    Build allocationSpreadsheet, weight optimization — 30+ min
  • 6
    Second opinionsFriends, Reddit, advisors — days
Pain Points

2-3 hours minimum

Even for a simple portfolio. Most people give up and buy an index fund.

No verification

Confirmation bias: you research stocks you already like. No adversarial challenge.

Stale by the time you finish

Earnings come out mid-research. News breaks. Your analysis is already outdated.

Our Solution

10 AI agents do all of this in parallel, with live web search, adversarial verification, and human approval — in under 2 minutes.

Agent 1

Intake Agent

Parses natural language into a typed investment profile

Input (Natural Language)

"I have $100k to invest. Moderate risk tolerance. Interested in AI and clean energy. 5-year horizon."

What It Handles

Ambiguous Risk Levels

"safe" → conservative, "balanced" → moderate, "growth" → aggressive

Sector Normalization

"tech" → Technology, "green" → Clean Energy, "crypto" → Cryptocurrency

Missing Fields

No budget? Default $10k. No horizon? Default 5yr. Always produces complete profile.

Output (Structured JSON)
{ "budget": 100000, "riskTolerance": "moderate", "sectors": [ "Artificial Intelligence", "Clean Energy" ], "timeHorizon": 5, "constraints": [], "summary": "Moderate-risk investor with $100k targeting AI and clean energy over 5 years" }
Why This Matters

Typed output means downstream agents get clean, validated input. No parsing prose. Schema enforcement catches malformed data before it propagates.

Agents 2-5

Research Phase: 4 Agents in Parallel

Each specialist searches the live web and returns structured findings

Market Research Agent

Web search: macro trends, sector outlook

Searches for current GDP, interest rate environment, sector-specific growth drivers and headwinds. Rates each sector's trend (strong growth / stable / declining) with confidence level.

Output: outlook, sectors[], macroFactors[], risks[]

Stock Screener Agent

Web search: equities by fundamentals

Screens for P/E < 35, revenue growth > 10%, market cap > $5B. Adjusts for sector norms. Returns 12-15 real ticker candidates with approximate valuations and one-line rationale each.

Output: candidates[{ticker, sector, peRatio, growth, reason}]

Risk Analyst Agent

Web search: volatility, correlation data

Assesses overall risk level, identifies specific risk factors with severity ratings. Analyzes correlation between target sectors. Estimates worst-case drawdown scenario.

Output: overallRisk, factors[], correlationNotes, maxDrawdown

News Sentiment Agent

Web search: headlines, earnings, red flags

Scans news from the last 30 days. Assesses sentiment per sector with top headlines. Flags red flags: pending earnings, regulatory changes, insider activity, supply chain issues.

Output: overallSentiment, sectorSentiment[], redFlags[]

All 4 run simultaneously via parallel() — wall-clock time = slowest agent (~40s), not sum (160s)

Agents 7a-7c

Adversarial Verification

Three independent skeptics challenge the portfolio. Default: flag if uncertain.

Skeptic #1
Concentration Risk
Checks:
• Any single holding > 15%?
• Any sector > 30%?
• Top 3 combined > 40%?
• Correlated sectors > 50%?
• Risk tolerance match?
Example Verdict
FLAG (medium): Semis at 27%
Skeptic #2
Sector & Timing Bias
Checks:
• Chasing recent returns?
• All consensus favorites?
• Policy-dependent bets?
• Survive a 2022 drawdown?
• Defensive positioning?
Example Verdict
PASS: Good sector spread
Skeptic #3
Data Freshness
Checks:
• Pending earnings?
• Pending regulation?
• Stale valuation data?
• Red flags ignored?
• Major upcoming events?
Example Verdict
FLAG (low): SEDG earnings pending
Why "Default to Flagging"?

LLMs have a politeness bias. Without explicit instruction to be skeptical, a reviewer will say "looks good" to avoid conflict. The adversarial framing forces genuine challenge.

Self-Healing Loop

If any skeptic flags an issue, a Revision Agent fixes the portfolio and re-runs verification. Max 2 rounds. This is the loop-until-done pattern in action.

Code

The Full Orchestration

~60 lines of JavaScript orchestrate 10 agents end to end

Research Fan-Out
phase('Intake') const profile = await agent( intakePrompt(args.query), { schema: PROFILE_SCHEMA } ) phase('Research') const [market, stocks, risk, news] = await parallel([ () => agent(marketPrompt(profile), { label: 'market', schema: MARKET_SCHEMA }), () => agent(screenerPrompt(profile), { label: 'screener', schema: STOCKS_SCHEMA }), () => agent(riskPrompt(profile), { label: 'risk', schema: RISK_SCHEMA }), () => agent(newsPrompt(profile), { label: 'news', schema: NEWS_SCHEMA }), ])
Build + Verify Loop
phase('Build Portfolio') let portfolio = await agent( buildPrompt(profile, market, stocks, risk, news), { schema: PORTFOLIO_SCHEMA } ) phase('Verify') let round = 0 while (round < 2) { const verdicts = await parallel([ () => agent(skepticPrompt( 'concentration', portfolio)), () => agent(skepticPrompt( 'bias', portfolio)), () => agent(skepticPrompt( 'freshness', portfolio)), ]) const issues = verdicts .filter(v => v?.hasIssue) if (!issues.length) break portfolio = await agent( revisePrompt(portfolio, issues)) round++ }

Design

Orchestration Patterns in This System

Five patterns composed into one coherent pipeline

Pattern Where Used Why Agents
SEQUENTIAL Intake → Research → Build Each phase depends on the previous phase's output 3 phases
FAN-OUT / FAN-IN 4 research agents in parallel Research tasks are independent; parallelism cuts time 4x 4 parallel
ADVERSARIAL 3 skeptics challenge portfolio Catches concentration, bias, and stale data that builders miss 3 parallel

How to Run It

One command in Claude Code launches the full multi-agent pipeline

Invoke the Workflow
Workflow({ scriptPath: 'investment-advisor.js', args: { query: "I have $100k to invest. Moderate risk. Interested in AI and clean energy. 5-year horizon." } })
What Happens
  • 1
    Intake agent parses your query — extracts budget, risk, sectors, horizon into typed JSON (~10s)
  • 2
    4 research agents fan out in parallel — each searches the web for live market data (~40s)
  • 3
    Portfolio builder synthesizes all research — produces 10 holdings with weights and rationale (~20s)
  • 4
    3 skeptics verify adversarially — flags get auto-fixed, max 2 revision rounds (~30s)
  • 5
    Final portfolio returned with full audit trail — profile, research, portfolio, verdicts, metadata

Total time: ~90 seconds — vs 2-3 hours manually

Important

This Is a Demo, Not Financial Advice

What you'd need to make it production-ready

This workflow is a technical demonstration of multi-agent orchestration patterns. It does not constitute investment advice and should not be used for real financial decisions.

What This Demo Does Not Do
  • ✗ Know your existing portfolio or financial situation
  • ✗ Use verified financial data APIs (uses web search only)
  • ✗ Guarantee accuracy of any numbers (LLMs can hallucinate)
  • ✗ Account for taxes, fees, or transaction costs
  • ✗ Backtest against historical market data
What Production Would Require
  • ✓ Real data APIs (Polygon.io, IEX Cloud, Bloomberg)
  • ✓ User portfolio input (brokerage CSV or Plaid)
  • ✓ Backtesting engine for portfolio validation
  • ✓ Regulatory compliance (RIA registration, disclaimers)
  • ✓ Deterministic fact-checking layer on LLM outputs
Data Source Reality

Web search returns unstructured, unverified data. Real financial products need timestamped feeds from licensed providers. Free options exist (Alpha Vantage, Yahoo Finance API) but have rate limits and no SLA.

The Real Value

The patterns demonstrated here (fan-out research, adversarial verification, revision loops) are production-ready. The data pipeline is what needs replacing. The orchestration is the reusable part.
